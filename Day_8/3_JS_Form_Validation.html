"submit"
The submit Event
Fires on the <form> element itself (not the button) when the form is submitted — by clicking the submit button OR pressing Enter in a field.
Always attach the listener to the form, not the button. This way Enter-key submission is caught too, not just button clicks.
event
The Event Object
The browser automatically passes an "event" object into your callback function. It contains details about what just happened.
You can name this parameter anything (e, evt, event) — it's just a regular function parameter. "event" is the clearest name for beginners.
.preventDefault()
Cancel the Default Action
Tells the browser "don't do what you'd normally do for this event." For forms, that means: don't reload the page.
preventDefault() is used for many events: stop a link from navigating, stop right-click menus, stop form submission. It must be called inside the event handler, every time.
This must be the FIRST line inside your submit handler. If you put validation checks before preventDefault(), the page might reload before your alert/error message even shows.
Step 1
Validate Name
The simplest possible check — is the field empty?
the code
script.js — inside submit handler
1 let name = document.getElementById("name").value;
2
3 if(name == "") {
4 alert("Name Required");
5 }
.value
Reading the Input's Content
.value gets whatever text the user has typed into an <input> field. This is different from innerHTML which is for displaying content, not reading form inputs.
innerHTML reads/writes content inside an element's tags. .value reads/writes the current text inside an <input>, <textarea>, or <select>. Mixing these up is the #1 beginner mistake.
name == ""
Empty String Check
"" is an empty string — zero characters. If the user typed nothing, name will equal "" exactly.
We'll upgrade this to === (strict equality) in the complete version — best practice in real projects to avoid type coercion bugs.
try it — type a name (or leave empty) and submit
This single check works — but using alert() for every error is annoying for users. In the complete version, we'll display the error directly on the page instead.
Step 2
Validate Email
Exact same pattern as name — just a new field id
the code
script.js — inside submit handler
1 let email = document.getElementById("email").value;
2
3 if(email == "") {
4 alert("Email Required");
5 }
Notice the pattern repeats exactly. get the value → check if empty → show an error. This is the core validation pattern you'll use for almost every field.
try it — type an email and submit
This basic check only confirms the field isn't empty — it doesn't verify the email looks like a real email (has @ and a domain). That's a more advanced check using regular expressions, which we'll touch on in later lessons.
Step 3
Validate Password
A different kind of check — not just "empty", but "long enough"
the code
script.js — inside submit handler
1 let password = document.getElementById("password").value;
2
3 if(password.length < 6) {
4 alert("Password should contain
5 at least 6 characters");
6 }
password.length
String Length Property
.length counts the number of characters in a string. "abc".length is 3. password.length tells us how many characters the user typed.
.length is a property, not a method — no parentheses! Common mistake: writing .length() instead of .length.
< 6
Minimum Length Check
If the password has FEWER than 6 characters, it's too short and fails validation. 6 characters or more passes this check.
Real-world password rules often require even more: uppercase, numbers, symbols. We're keeping it simple with a length check for this lesson.
try it — type a password and submit
0 characters
Notice: 3 separate if-blocks means 3 separate alert popups could appear back-to-back if multiple fields are wrong. That's a poor user experience — the complete version fixes this with return.

Deep Dive
Explain Line by Line — 7 Concepts to Master
Click each concept — students must understand all of these to fully grasp the validation
addEventListener()
1. addEventListener()
Attaches a function to run when a specific event happens on an element. Here it's attached to the <form>, listening for "submit".
Pattern: element.addEventListener("eventType", callbackFunction). This is the same pattern from Hour 3 of Day 7 — now applied to forms specifically.
"submit"
2. submit (the event type)
The event that fires when a form is submitted — either by clicking a submit button or pressing Enter inside a text field.
Always listen on the <form> element for "submit", not on the button for "click". This catches BOTH submission methods.
preventDefault()
3. preventDefault()
Stops the browser's default action — for a form submit event, that default action is "reload the page".
Without this line, even if validation passes, the page would reload and any JS-driven success message would instantly disappear.
trim()
4. trim()
Removes whitespace from the start and end of a string. " John " becomes "John". Prevents users from "cheating" empty validation by typing just spaces.
Without trim(), a user could type " " (just spaces) into the name field and it would NOT equal "" — so the empty check would wrongly pass.
value
5. value
Reads the current text inside an input field. Different from innerHTML, which reads content between HTML tags rather than form input data.
.value works on <input>, <textarea>, and <select>. It's always a string, even for type="number" inputs — you may need Number() to convert it.
length
6. length
A property (not a method!) that tells you how many characters are in a string. Used here to enforce a minimum password length.
"hello".length is 5. No parentheses. This is one of the most common JS properties used across strings and arrays alike.
return
7. return
Immediately exits the function. Used here to stop running the rest of the validation checks once one error is found — prevents multiple errors overwriting each other.
Without return, if name is empty, the code would still continue to check email and password, and only the LAST error message would be visible — confusing for users. return makes validation stop at the first problem found.
Teaching tip: Have students predict what happens if you remove the return statement from one if-block. Then test it live — watch how the message flickers through multiple errors before landing on the last one.

+
String Concatenation
The + operator joins strings together. "Welcome " + name combines the fixed text with the variable's value into one new string.
You can mix HTML tags right inside the strings — like "<br>" for a line break — since innerHTML interprets HTML, not just plain text.
"<strong>"
HTML Inside innerHTML
Because we're setting .innerHTML (not .textContent), any HTML tags we include — like <br> or <strong> — actually render as real HTML elements.
This is exactly why innerHTML is powerful — and also why you should never put raw, untrusted user input into innerHTML without sanitising it (a security concept called XSS, covered in later weeks).

hint

Inline Error Display — Bootstrap Pattern
Production apps show errors next to the field, not in a single message div — here's how
html — bootstrap invalid-feedback pattern
index.html
<input type="text" id="mobile"
class="form-control">
<div id="mobileError"
class="invalid-feedback"></div>
javascript — toggle classes + set message
script.js
if(mobile.length !== 10) {
document.getElementById("mobile")
.classList.add("is-invalid");
document.getElementById("mobileError")
.innerHTML = "Must be 10 digits";
return;
}
live preview — try an invalid mobile number
Mobile number must be 10 digits
classList.add()
Add a CSS Class via JS
Adds a class name to an element without removing existing ones. Here, "is-invalid" triggers a red border style via CSS.
classList also has .remove(), .toggle(), and .contains() — useful for clearing errors when the user fixes their input.
This per-field pattern is more work to write, but gives users a much clearer experience than one shared #message div — they instantly know which field has the problem.
