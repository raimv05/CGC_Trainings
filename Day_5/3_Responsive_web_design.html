padding:30px
Halved from 60px → 30px
Desktop sections have generous 60px padding. Mobile screens can't afford that — 30px reclaims valuable vertical space.
Rule: desktop padding ÷ 2 = a safe starting mobile value. Test and adjust as needed.
font-size:26px
Heading shrinks from ~40px → 26px
A 40px heading on a 360px-wide phone is visually overwhelming and may wrap to 4+ lines. 26px fits cleanly in 1–2 lines.
Good mobile font-size scale: h1 = 26–30px, h2 = 22–26px, p = 14–16px, small = 12–13px.
width:100%
Full-width button ⭐
On mobile, buttons should span the full container width. This is a standard mobile UX pattern — larger tap target, cleaner look.
Apps like WhatsApp, Instagram, Google all use full-width primary buttons on their mobile login screens. It signals "this is the main action."

Bonus Topic
Responsive Units — px vs % vs rem
The difference between fixed and flexible measurements — a key senior-developer concept
The Rule: avoid hardcoded px for widths and font-sizes wherever possible.
Use % for widths, rem for font-sizes. Reserve px only for small fixed values like borders, shadows, and spacing where you want exact control.
📌
px — Fixed
Always exactly that many pixels. Ignores the user's browser font preferences and the parent container's size.
div { width: 400px; }
/* always 400px — even on a 360px phone */
⚠ Can overflow on small screens
🔧
% — Flexible
A percentage of the parent element's width. Automatically adjusts when the container or screen resizes.
div { width: 80%; }
/* 80% of its parent — always fits */
✅ Adapts to any screen width
📐
rem — Scalable
Relative to the root font size (usually 16px). If the user zooms their browser, everything scales proportionally.
h2 { font-size: 2rem; }
/* 2 × 16px = 32px by default */
✅ Respects user accessibility settings
practical upgrade for skillhub
instead of px, use % or rem
/* ❌ Fixed — may overflow */
.feature { width: 400px; }
/* ✅ Flexible — always fits */
.feature { width: 80%; }
/* ❌ Fixed font — doesn't respect zoom */
body { font-size: 16px; }
/* ✅ Scalable — zoom-friendly */
body { font-size: 1rem; }

Git Milestone
Commit Your Responsive Website to Git
Real developers save meaningful milestones — this is your first one
Why commit now?
You've just completed something meaningful — a fully responsive landing page. This is the right time to save your progress. If you break something tomorrow while adding JavaScript, you can always return to this working version.
run these 3 commands in your terminal
Terminal — in your project folder
$ git add .
← stages all changed files
$ git commit -m "Completed responsive landing page"
← saves a snapshot with your message
$ git push origin main
← uploads to GitHub (if repo exists)
what each command does
git add .
Stage all changes
The dot (.) means "all files". Git marks every new or changed file to be included in the next commit.
You can also add specific files: git add style.css. The dot is a shortcut for "everything in this folder and all subfolders."
git commit -m
Save a snapshot
Creates a permanent checkpoint with the message you write. The -m flag lets you add the message inline.
Good commit messages are short and specific: "Added responsive hero section" not "fixed stuff". This becomes your project history.
git push origin main
Upload to GitHub
Sends your committed snapshots to the remote repository on GitHub.com so your code is backed up online.
origin = the GitHub remote. main = the branch name. If your branch is called master, use git push origin master.
Commit after every meaningful milestone. A good rule of thumb: if you'd be annoyed to lose this work, it's time to commit. Start building this habit now — it becomes second nature by Week 3.
