Why Do We Need "Update"?
A student registers with a typo in their email. What now?
Rahul registers as "rahul@gmial.com" by mistake. How do we fix it right now?
the only option right now
deleteStudent(3); // remove the student entirely
// ...then they fill the ENTIRE form again

Terrible user experience.
Deleting and re-registering loses the original registration date, risks the admin mis-typing something else, and frustrates the student.
We need a way to edit a record in place — that's the difference between Delete and Update.

Part 2 · 20 min
editStudent() & the editIndex Variable
How JavaScript remembers WHICH student you're editing
Add this near the top of script.js, then below your functions
frontend/js/script.js
let editIndex = -1; // -1 = "not editing, adding new"
function editStudent(index) {
editIndex = index;
document.getElementById("name").value =
students[index].name;
document.getElementById("email").value =
students[index].email;
document.getElementById("course").value =
students[index].course;
document.getElementById("submitBtn").textContent = "Update Student";
}
why a single number, -1?
editIndex is a "memory slot" that holds either:

-1 → no student is being edited, the form is for a NEW registration.

0, 1, 2… → the exact position in students[] currently loaded into the form.
after clicking edit on row 1
Console
▶editStudent(1)
▶editIndex: 1
Note: this only fills the form — it doesn't change students[] yet. The actual update happens on Submit, in the next step.
Part 2 · continued
The Submit Handler — Create OR Update
One function, two jobs: add a NEW student, or REPLACE an existing one
frontend/js/script.js — registerStudent()
function registerStudent(e) {
e.preventDefault();
let student = {
name: document.getElementById("name").value,
email: document.getElementById("email").value,
course: document.getElementById("course").value
};
if(editIndex == -1) {
students.push(student); // CREATE
} else {
students[editIndex] = student; // UPDATE
editIndex = -1;
}
saveData(); // localStorage.setItem(...)
displayStudents();
document.getElementById("regForm").reset();
document.getElementById("submitBtn").textContent = "Register Student";
}
1
Readthe form values into a fresh student object
2
DecideeditIndex == -1 ? push() a new one : replace the existing one
3
Savewrite the updated array back to localStorage
4
Rendercall displayStudents() again so the table reflects reality
students[editIndex] = student; replaces the WHOLE object at that array position in one line — no need to update name, email, and course separately. The object at that index is simply swapped out.
Part 3 · 10 min
Why Do We Need Search?
SkillHub now has 200 registered students. Finding "Priya" means...
With 200 rows in the table, how does an admin find one specific student?
Not by scrolling.
We need to type a few letters and instantly narrow the list down to matching records.
That's Search — and JavaScript's array filter() method does exactly this.
Slow
Scrolling through 200 rows by eye takes real time
Error-Prone
Easy to miss the row you're actually looking for
Doesn't Scale
What happens at 2,000 students instead of 200?
Concept
Live Search with filter()
Every keystroke narrows the array down to matching records
Add this below your other functions
frontend/js/script.js
document.getElementById("searchBox").addEventListener("keyup", function() {
let keyword = this.value.toLowerCase();
let filtered = students.filter(function(s) {
return s.name.toLowerCase().includes(keyword);
});
displayStudents(filtered);
});
what filter() does
students.filter()
↓
checks EVERY student
↓
keeps only the matches
↓
returns a brand-new array
Console
▶filtered = students.filter(...)
▶[{name: "Priya", ...}]
Key idea: filter() never changes the original array — it returns a brand-new one with just the matches. students itself stays completely untouched.
try it — type a name below
// 8 sample students — matches: 8 of 8
Gotcha: in this simple version, Edit/Delete buttons on a FILTERED list still carry the index from the ORIGINAL array — so they can point at the wrong row. Production apps usually search by a unique id instead of a raw array index, which we'll explore later in the course.

Part 4 · continued
Calculating the Stats with filter()
The same method we used for search — counting instead of filtering rows
frontend/js/script.js
function updateStats() {
document.getElementById("totalCount").textContent = students.length;
document.getElementById("mernCount").textContent =
students.filter(s => s.course === "MERN").length;
document.getElementById("javaCount").textContent =
students.filter(s => s.course === "Java").length;
document.getElementById("pythonCount").textContent =
students.filter(s => s.course === "Python").length;
}
the pattern
filter() keeps only the matching students.
.length turns that filtered array into a single number.

Call updateStats() at the end of displayStudents() so the cards refresh automatically — every add, edit, delete, and search keeps the dashboard honest.
Remember: length is a property, not a method — students.filter(...).length, no parentheses after length. The same rule from Day 9's first 90 minutes still applies.

Line-by-Line — Update, Search & Stats
Click any row to expand — every concept from this 90 minutes
editIndex = -1
editIndex — "what am I editing?"
A variable that tracks whether the form is creating a new record or updating an old one
-1 means "add mode." Any other number is the exact index of the student currently loaded into the form. It gets reset to -1 right after a successful update.
editStudent(i)
editStudent(index) — load a record into the form
Sets editIndex and copies the student's existing data into the input fields
This function never touches the students array. It only reads from it — the actual write happens later, in registerStudent().
students[i] = s
students[editIndex] = student — overwrite in place
Replaces the object at that position with the freshly-edited one
Unlike push() (which adds a NEW item), assigning to an existing index swaps out the object that's already there. The array's length doesn't change.
.filter(cb)
filter() — keep only what matches
Runs a test function against every item and returns a NEW array of the ones that passed
Used for search (name includes keyword?) AND for statistics (course === "MERN"?). Same method, two different jobs — just a different test inside.
"keyup"
keyup event — fires on every key release
Triggers the search function instantly as the admin types, without needing a Submit button
this.value inside the listener refers to the search input itself — "this" is the element the event happened on.
confirm()
confirm() — ask before destroying data
Shows a native browser popup with OK/Cancel and returns true or false
Used before splice() (delete one) and before clearing the entire array (Clear All) — exactly what an if() statement needs to guard a destructive action.
