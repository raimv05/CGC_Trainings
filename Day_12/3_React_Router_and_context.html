Why React Router?
3 Hours · React Router Fundamentals
Project Goal
Convert SkillHub LMS from a Single Component into a Multi-Page Application.
start with a question
Currently, everything is inside App.jsx. What happens if our project has 50 pages? Impossible to maintain. We need routing.
install
terminal
$ npm install react-router-dom
wrap the application — main.jsx
main.jsx
import { BrowserRouter } from "react-router-dom";

ReactDOM.createRoot(document.getElementById("root")).render(
  <BrowserRouter>
    <App/>
  </BrowserRouter>
);
BrowserRouter controls navigation — it listens to the URL and decides what to render, without ever asking the browser for a fresh page.

Dynamic Routes & the 404 Page
One route template, many URLs — plus a catch-all for the rest

the 404 catch-all
App.jsx
<Route path="*" element={<NotFound/>} />
The "*" path matches any URL that didn't match an earlier route — always place it last, so it acts as a true catch-all.

Day 25 · Part 1
The Prop Drilling Problem
3 Hours · Context API & Authentication State Management
Project Goal
Implement Authentication State using Context API.
the current problem
Navbar needs the user's name. Dashboard needs it. Profile needs it too. Passing it down manually means threading one prop through every layer in between:
App ↓ Navbar ↓ Dashboard ↓ Profile ↓ Settings
Prop Drilling: components in the middle (Dashboard, Profile) have to accept and pass along a prop they never actually use themselves, just so it reaches the bottom.
Solution — Context API: put the data in one shared place that every component can reach directly, no matter how deep it is.
Day 25 · Part 1
Context API — createContext, Provider & useContext
One shared value, reachable from anywhere
AuthContext.Provider (value = user)
↓
Navbar
↓
Dashboard
↓
Profile — useContext(AuthContext)
create context — context/AuthContext.jsx
AuthContext.jsx
import { createContext } from "react";
export const AuthContext = createContext();

// wrap the app
<AuthContext.Provider value={{user}}>
  <App/>
</AuthContext.Provider>

// read it, anywhere
const {user} = useContext(AuthContext);
<h2>Welcome {user}</h2>
Students now understand Global State — one Provider at the top, useContext anywhere below it, no manual prop passing required.

Day 25 · Part 2
Protected Route
Dashboard should be unreachable without logging in first
ProtectedRoute.jsx
function ProtectedRoute({children}){
  if(!user){
    return <Navigate to="/login" />
  }
  return children;
}

// wrap Dashboard
<ProtectedRoute>
  <Dashboard/>
</ProtectedRoute>
try it live — same login state as the previous slide

Day 25 · Part 2
Session Persistence
Stay logged in even after refreshing the browser
App.jsx
// on login
localStorage.setItem("user", JSON.stringify(userData));

// restore on load
useEffect(() => {
  const saved = localStorage.getItem("user");
  if(saved) setUser(JSON.parse(saved));
}, []);
try it live
localStorage is empty
Students remain logged in after refresh — exactly the same "save it → load it" pattern used for Local Storage and students.json earlier in the course.
