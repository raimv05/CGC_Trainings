What we need: Browser Storage — a place inside the browser itself that keeps data even after a refresh, a tab close, or a computer restart. That's exactly what localStorage is for.

Problem
The JSON Problem
Our students variable is an Array of Objects. Can localStorage save that directly?
Can Local Storage save an Array (like our students list) the same way it saved "Manish"?
the wrong way
localStorage.setItem("students", students);
console.log(localStorage.getItem("students"));
// "[object Object],[object Object]" ← data destroyed!
No.
localStorage can only store plain text (strings). Give it an Array or Object and JavaScript silently converts it to the useless text "[object Object]" — all the real data is lost.
We need to convert our data into a text format first: JSON.
saving — JSON.stringify()
frontend/js/script.js
localStorage.setItem(
"students",
JSON.stringify(students)
);
Turns the Array of Objects into a plain text string that localStorage can safely store.
loading — JSON.parse()
frontend/js/script.js
let students =
JSON.parse(
localStorage.getItem("students")
) || [];
Turns the text string back into a real, usable Array of Objects.
the round trip
Object / Array JSON.stringify() Text in localStorage JSON.parse() Object / Array again
|| [] matters: The very first time the page loads, getItem("students") returns null — nothing has been saved yet. JSON.parse(null) would give null, and calling .push() on null crashes. || [] falls back to an empty array so the app never breaks.

Explain
Line-by-Line — Local Storage & JSON
Click any row to expand — every piece of syntax from Part 1
localStorage
localStorage — the browser's storage object
A built-in object every browser gives you, for free, to store text data
Data saved here belongs to this website only (other sites can't read it) and stays until it's explicitly removed or the browser storage is cleared — refreshing or closing the tab does NOT delete it.
.setItem(k,v)
setItem(key, value) — save data
Stores a value under a named key, overwriting any old value for that key
Both key and value must be strings. Calling setItem("students", ...) a second time completely replaces the first save — it doesn't add to it.
.getItem(k)
getItem(key) — read data
Returns the string stored under that key, or null if nothing was ever saved
This is a read-only lookup — it doesn't remove the data. You'll typically wrap it in JSON.parse() when the value is an array or object.
JSON.stringify
JSON.stringify(value) — object/array → text
Converts a JavaScript array or object into a JSON-formatted string
This is the ONLY safe way to put an array or object into localStorage. Without it, JavaScript converts your data to the useless text "[object Object]".
JSON.parse
JSON.parse(string) — text → object/array
Converts a JSON-formatted string back into a real JavaScript array or object
The exact reverse of stringify(). You need this every time you read array/object data back out of localStorage — otherwise you just get plain text you can't push() or loop through.
|| []
|| [] — safe fallback
If the left side is null, undefined, or falsy, use an empty array instead
On the very first visit, localStorage has nothing saved yet, so getItem() returns null. || [] catches that case so students always starts as a usable array, never null.
.removeItem(k)
removeItem(key) — delete one key
Permanently removes that key and its value from storage
We'll use this later today for the "Clear All" button — removeItem("students") wipes the entire saved dashboard in one line.

editIndex is the whole trick: it starts at -1, meaning "not editing anyone." Clicking Edit sets it to that student's position in the array. The submit button will check this value to decide whether to add a new student or update an existing one — that's next.
Build · 20 min
One Submit Button, Two Jobs
Add a NEW student, or UPDATE the one being edited — same button decides which
Update inside your submit handler
inside registerStudent() / form submit
// name, email, course read from the form as usual
let student = { name, email, course };
if (editIndex == -1) {
students.push(student); // new student
} else {
students[editIndex] = student; // overwrite existing
editIndex = -1; // reset — back to "add" mode
}
localStorage.setItem("students", JSON.stringify(students));
displayStudents();
form.reset();
editIndex == -1 → ADD
The form is in its normal state. push() adds the new student object as one more item at the end of the array.
editIndex != -1 → UPDATE
The form was pre-filled by editStudent(). Instead of adding a new item, we overwrite students[editIndex] with the edited values.
Don't forget to reset editIndex back to -1 after an update. If you skip this, the very next "new" registration will silently overwrite the student you just edited instead of adding a fresh one.
Same button, both jobs. Every save — new or edited — runs through the same localStorage.setItem line, so persistence keeps working automatically. No extra code needed for that part.

Flow
The Full Persistence Journey
From typing in a form to surviving a browser refresh — every step, in order
1
Form → Object → Array
The student types into the form. JavaScript builds an object and pushes it into the students array — exactly what Day 9 already built.
2
Array → JSON.stringify() → Text
Right after the push, the whole array is converted into one long text string using JSON.stringify().
3
Text → localStorage.setItem()
That text string is saved into the browser's localStorage under the key "students" — this is what makes it durable.
4
↓ Refresh happens
The browser reloads. RAM is wiped. Every variable, including students, resets. But localStorage is untouched — it lives outside RAM.
5
localStorage.getItem() → JSON.parse() → Array
At the top of script.js, the saved text is read back out and converted back into a real array with JSON.parse().
6
Array → displayStudents() → Table
The restored array is looped through with forEach() and rendered — the dashboard looks exactly like it did before the refresh.
Why this matters: This save → stringify → store → refresh → parse → restore cycle is the exact same pattern every real app uses to keep your session alive — the only difference is that production apps save to a database (MongoDB, later in this course) instead of localStorage.
